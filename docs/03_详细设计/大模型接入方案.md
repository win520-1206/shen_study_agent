---

## 1. 背景与目标

### 1.1 当前状态

LearnMate-AI 系统当前采用 **规则式智能体**，即 6 个智能体（画像、诊断、资源规划、内容生成、路径规划、审查）全部通过关键词匹配、模板拼接和条件判断来实现个性化输出。

规则式方案的优势是稳定、可控、无 API 成本；但存在明显局限：
- 关键词匹配无法理解否定句和复杂意图（例如"我不想一开始就上很多代码"会被误判为"喜欢代码实战"）
- 内容生成依赖模板，个性化深度有限
- 无法处理知识库未覆盖的自由提问

### 1.2 接入目标

将系统从"纯规则式"平滑升级为"大模型增强式"，实现：
1. **更准确的画像抽取**：理解自然语言中的意图、否定、转折
2. **更个性化的内容生成**：根据画像动态生成讲解、题目、案例，而非模板拼接
3. **一行配置切换**：通过 `USE_LLM` 开关在规则式和大模型模式间切换
4. **前端零改动**：数据结构完全兼容，前端无需任何修改

## 2. 设计原则

### 2.1 局部替换式升级

不重构整个系统，而是逐个替换智能体的实现。每个智能体都遵守统一接口：

```python
@dataclass
class AgentResult:
    name: str           # 智能体名称
    input_summary: str  # 输入摘要
    output_summary: str # 输出摘要
    payload: dict       # 业务数据
```

无论内部是规则匹配还是大模型推理，只要返回 `AgentResult`，编排器就无需任何改动。

### 2.2 优先级策略

| 优先级 | 智能体 | 原因 |
|--------|--------|------|
| P0（已实现） | 画像智能体 | 入口环节，画像质量决定后续所有输出 |
| P0（已实现） | 内容生成智能体 | 用户直接感知的内容，个性化价值最高 |
| P1（计划中） | 诊断智能体 | 需要综合分析能力，LLM 有优势 |
| P2（暂不替换） | 路径规划、资源规划、审查 | 逻辑相对固定，规则式已够用 |

### 2.3 知识库作为上下文（RAG 思想）

大模型不凭空生成内容，而是基于课程知识库提供上下文：

```
用户输入 → 画像抽取 → 知识库检索 → 将检索结果 + 画像 作为 LLM 提示词 → 生成内容
```

这样做的好处：
- 生成内容有据可查，减少幻觉
- 课程范围可控，不会生成无关内容
- 可以追溯每条资源的知识来源

│  │ [可切换]  │  │ [规则式]  │  │  [规则式]    │   │
│  └──────────┘  └──────────┘  └──────────────┘   │
│                                                   │
│  ┌──────────────┐  ┌──────────┐  ┌──────────┐   │
│  │内容生成智能体 │  │路径规划  │  │ 审查智能体│   │
│  │  [可切换]    │  │[规则式]  │  │ [规则式]  │   │

│  │  ┌──────────┐  ┌──────────────────────┐ │     │

│  └─────────────────────────────────────────┘     │
└─────────────────────────────────────────────────┘

### 3.2 配置驱动切换

在 `config.py` 中通过配置项控制：

```python
# 设为 True 启用大模型智能体，设为 False 使用规则式智能体
USE_LLM = False

# LLM 服务提供者: "mock" / "xfyun"
LLM_PROVIDER = "mock"

# 科大讯飞 API 配置
XFYUN_APP_ID = ""
XFYUN_API_KEY = ""
XFYUN_API_SECRET = ""
```

编排器根据 `USE_LLM` 自动选择实现：

```python
if USE_LLM:
    llm = get_llm_service()
    self.profile_agent = LLMProfileAgent(llm)
    self.content_generator_agent = LLMContentGeneratorAgent(llm)
else:
    self.profile_agent = ProfileAgent()
    self.content_generator_agent = ContentGeneratorAgent()
```

### 3.3 LLM 服务层抽象

```
LLMService (基类)
├── chat(system_prompt, user_prompt) -> str
└── chat_json(system_prompt, user_prompt) -> dict

MockLLMService          # 开发测试用，不调用真实 API
XfyunLLMService         # 科大讯飞星火大模型
```

`chat_json` 方法自动处理 JSON 解析，支持从 LLM 返回的文本中提取 JSON 对象。

## 4. 接入科大讯飞 API 方案

### 4.1 接入步骤

1. 注册科大讯飞开放平台账号，创建应用，获取 `APP_ID`、`API_KEY`、`API_SECRET`
2. 安装 SDK：`pip install spark-ai-python` 或使用 WebSocket 原生调用
3. 在 `config.py` 中填入真实凭证
4. 将 `USE_LLM` 改为 `True`，`LLM_PROVIDER` 改为 `"xfyun"`
5. 重启后端服务

### 4.2 Prompt 设计要点

**画像智能体 Prompt：**
- 系统角色：学习画像分析助手
- 输入：学生自然语言描述
- 输出要求：严格返回 8 维 JSON，每个维度限定可选值
- 知识库上下文：无需

**内容生成智能体 Prompt：**
- 系统角色：个性化学习助手
- 输入：学生画像 + 知识库检索结果（前 2000 字）
- 输出要求：严格返回 5 类资源的 JSON 数组
- 知识库上下文：必须提供当前模块内容

### 4.3 成本控制策略

- 单次完整流程（6 个智能体）中，仅 2 个调用 LLM
- 画像抽取：约 500 token 输入 + 200 token 输出
- 内容生成：约 3000 token 输入 + 2000 token 输出
- 预估单次调用成本：约 0.01-0.03 元（取决于讯飞定价）

## 5. 数据兼容性

### 5.1 不变部分

以下组件在接入 LLM 后 **完全不需要修改**：

| 组件 | 原因 |
|------|------|
| 数据库表结构 | `profile_json`、`diagnosis_json` 等字段存储的是 JSON 字符串，与生成方式无关 |
| API 接口定义 | 请求/响应的 Pydantic schema 不变 |
| 前端页面 | 消费的是同一套 API 和数据结构 |
| 知识库数据 | 作为 LLM 的上下文提供，本身不变 |
| 测试用例 | 接口行为不变，测试无需修改 |

### 5.2 变化部分

| 变化点 | 说明 |
|--------|------|
| `config.py` | 新增 `USE_LLM`、`LLM_PROVIDER`、API 凭证配置 |
| `orchestrator.py` | 根据配置动态选择智能体实现 |
| 新增 `llm_service.py` | LLM 服务抽象层 |
| 新增 `llm_agents.py` | LLM 版画像和内容生成智能体 |
| `AgentTrace` 记录 | LLM 模式的 trace 名称带 "(LLM)" 后缀，便于区分 |

## 8. 总结

本方案采用"局部替换式"策略，通过统一的 `AgentResult` 接口实现规则式和大模型式智能体的无缝切换。优先升级画像和内容生成两个核心智能体，通过 `USE_LLM` 配置开关控制，确保：

1. **前端零改动**：数据结构完全兼容
2. **数据库零改动**：JSON 字段存储与生成方式无关
3. **灵活切换**：一行配置切换模式
4. **渐进升级**：后续可逐步替换其他智能体
5. **成本可控**：仅核心环节调用 LLM，减少 API 开销
