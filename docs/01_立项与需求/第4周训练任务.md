# 第 4 周训练任务

第 1 周的关键词是"理解"。  
第 2 周的关键词是"参与"。  
第 3 周的关键词是"升级"。  
第 4 周的关键词是：

**打磨 + 补分。**

这一周你要做两件大事：
1. 接入真实大模型 API，让系统从"规则式"正式升级为"AI 驱动"
2. 补上比赛加分项：智能答疑、学习效果评估、流式输出

同时开始完善比赛文档，为最后的 PPT 和视频做准备。

---

## 第 1 天：实现通用大模型 API 接入

### 要做什么

1. 实现一个通用的 `GenericLLMService`，支持 OpenAI 兼容格式的 API
2. 配置 `LLM_BASE_URL`、`LLM_API_KEY`、`LLM_MODEL` 三个参数
3. 支持接入 DeepSeek、通义千问、讯飞星火（OpenAI 模式）等任何兼容 API
4. 更新 `config.py` 和 `get_llm_service()` 工厂函数
5. 验证 Mock 模式和真实 API 模式可以一键切换

### 为什么做

比赛要求使用科大讯飞工具，但你目前没有 API 凭证。实现通用接口的好处：
- 现在就能用 DeepSeek 等免费/低价 API 测试
- 比赛时可以根据实际情况选择最合适的模型
- 代码结构更通用，不绑定某一家供应商

### 你会学到什么

- 什么是 OpenAI 兼容格式（Chat Completions API）
- 如何用 HTTP 调用大模型 API
- 什么是供应商无关的接口设计
- 如何管理 API 密钥和配置

### 完成标准

1. 改 3 个配置项就能切换到真实 API
2. Mock 模式和真实模式测试均通过
3. 你能解释"为什么用通用格式而不是直接对接讯飞 SDK"

## 第 2 天：Prompt 优化 + LLM 诊断智能体

### 要做什么

1. 优化 `LLMProfileAgent` 的 prompt：增加 few-shot 示例，提升抽取准确率
2. 优化 `LLMContentGeneratorAgent` 的 prompt：增加输出格式约束和质量要求
3. 新增 `LLMDiagnosisAgent`：用 LLM 替代规则式诊断逻辑
4. 更新编排器，支持诊断智能体也使用 LLM 模式
5. 对比优化前后的输出质量

### 为什么做

Prompt 质量直接决定 LLM 输出质量。好的 prompt 应该：
- 有明确的角色定义
- 有具体的输出格式要求
- 有 few-shot 示例（让模型知道"什么样的回答是对的"）
- 有约束条件（防止模型自由发挥）

### 你会学到什么

- 什么是 prompt engineering（提示词工程）
- few-shot learning 的作用
- 如何设计结构化输出的 prompt
- 如何迭代优化 prompt

### 完成标准

1. LLM 画像智能体能正确处理否定句（如"不想上很多代码"）
2. LLM 内容生成的质量明显优于 Mock 模板
3. LLM 诊断智能体的推荐策略比规则式更自然

---

## 第 3 天：智能答疑（课程知识问答）

### 要做什么

1. 新增 `QAAgent`：基于知识库回答课程相关问题
2. 新增 API 端点：`POST /api/v1/chat/qa`
3. 设计问答的数据流：用户提问 → 知识库检索 → LLM 生成回答
4. 新增前端组件：问答面板（可选，取决于时间）
5. 新增请求/响应 schema

### 为什么做

智能答疑是比赛的加分项，而且是评委最容易理解的功能——"学生遇到不懂的，可以直接问"。

### 你会学到什么

- RAG 问答的基本流程
- 如何设计一个新的 API 端点
- 知识库检索在问答场景中的作用
- 如何让回答"有据可查"

### 完成标准

1. POST `/api/v1/chat/qa` 能接收问题并返回回答
2. 回答基于知识库内容，不是凭空生成
3. 能处理"什么是线性回归"这类概念问题

## 第 4 天：学习效果评估增强

### 覀什么做

1. 增强 `submit_assessment` 接口：记录更多评估维度
2. 新增评估历史查询接口
3. 实现"评估结果影响后续推荐"的逻辑
4. 新增弱项追踪：根据多次评估结果识别持续薄弱的知识点
5. 前端新增评估历史展示区（可选）

### 为什么做

当前的评估只是"提交分数 → 返回反馈"，没有闭环。真正的学习系统应该：
- 记录每次练习的成绩
- 分析哪些知识点一直没掌握好
- 根据历史表现调整推荐策略

### 你会学到什么

- 什么是"反馈闭环"
- 如何用历史数据影响未来决策
- 数据聚合和趋势分析的基本思路

### 完成标准

1. 多次提交评估后，系统能识别"持续薄弱"的知识点
2. 二次对话时，推荐内容会参考历史评估结果
3. dashboard 接口返回评估历史和弱项分析

---

## 第 5 天：SSE 流式输出

### 要做什么

1. 后端新增 SSE（Server-Sent Events）流式接口
2. 前端用 EventSource 接收流式数据
3. 实现"打字机效果"逐字渲染
4. 重点在内容生成环节的流式展示
5. 保持非流式接口的兼容性

### 为什么做

流式输出是比赛演示的加分体验——评委可以看到内容"实时生成"，而不是等几秒后一次性弹出。这会让系统看起来更"智能"、更"实时"。

### 你会学到什么

- 什么是 SSE（Server-Sent Events）
- 流式输出和普通 HTTP 响应的区别
- 前端如何处理流式数据
- 如何在保持功能的同时增加流式体验

### 完成标准

1. 内容生成环节支持流式输出
2. 前端能逐字渲染生成内容
3. 非流式接口仍然正常工作
